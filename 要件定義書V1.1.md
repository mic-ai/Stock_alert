# 個別株スクリーニング&アラートシステム 要件定義書

## 0. 前提・重要な訂正
- 当初検討していた「LINE Notify」は2025年3月31日にサービス終了済み。
  本システムでは後継の **LINE Messaging API** を使用する。
- 個人が自分専用に通知を受け取る用途のため、LINE公式アカウントを自分で作成し、
  自分自身（または非公開のLINEグループ）にpushメッセージを送る構成とする。
  Messaging APIは月200通まで無料（個人利用の通知回数なら十分収まる想定）。

## 1. 目的
海外業種特化型指数（例: SOX等）と国内業種特化型指数（例: 日経半導体株指数）の
トレンドを毎日自動判定し、ワイコフメソッドの近似ルールに合致する銘柄を
「買い候補」「売り候補」としてLINEおよびメールにアラート送信する。
判定はすべてルールベース（閾値による決定的ロジック）で行い、
AIの都度推論には依存しない。
あわせて、過去のアラート（予測）が実際に的中したかどうかを追跡・集計し、
日次で的中率をメール通知する。

## 2. システム全体構成
```
[①データ取得層] → [②ルール判定エンジン層] → [③スケジューラ(GitHub Actions)] → [④通知層]
                                                              ↓
                                                    [⑤予測評価層(的中率トラッキング)]
```
- 実行環境: GitHub Actions（スケジュール実行 = cron構文）
- 言語: Python 3.x
- 主要ライブラリ: yfinance, pandas, pandas-ta, requests, smtplib（標準ライブラリ）

## 3. スケジュール仕様
| 時刻(JST) | 処理内容 |
|---|---|
| 5:30頃 | 米国業種特化型指数のトレンド判定（前日の米国市場クローズ後データを使用） |
| 8:00 | ①の結果を使い、上昇トレンド業種内の候補銘柄をワイコフ近似ルールでスクリーニング→買い候補アラート送信（LINE＋メール）、`predictions.csv`に記録 |
| 8:00 | 下降トレンド業種＋保有銘柄リストをワイコフ近似ルールで判定→売り候補アラート送信（LINE＋メール）、`predictions.csv`に記録 |
| 8:10頃 | **【追加】評価期限(N営業日後)が到来した過去の予測を判定し、的中率を集計。日次サマリーをメールで送信** |

- GitHub Actionsの `schedule: cron:` はUTC基準のため、JST変換時に夏時間・冬時間の概念がない点に注意
  （日本はDSTがないため一定のUTCオフセット+9時間で固定でよい。米国側の夏時間切替は
  「米国市場クローズ後のデータを使う」設計のため、こちらで時刻補正する必要はない）。

## 4. データ取得仕様

### 4-1. 米国業種特化型指数（トレンド判定用）
- 取得元: 海外データベンダー（既定：使用するベンダーのAPIキーは環境変数/GitHub Secretsで管理）
- 対象指数（例）: SOX, XLK, XLV, XLF, XLE, XLU等（別紙「業種特化型指数一覧」参照）

### 4-2. 個別銘柄データ（日本株・米国株、RSI・ワイコフ判定用）
- 取得元: **yfinance**（無料・APIキー不要、非公式ラッパーである点に留意）
- 日本株ティッカー: 証券コード+".T"（例: "6857.T"）
- 米国株ティッカー: そのまま（例: "NVDA"）
- 取得は20銘柄ずつバッチ処理し、バッチ間に1秒程度のインターバルを入れる
- 取得結果が空(`df.empty`)の場合はエラーではなく警告としてログに記録し、
  「データ欠損なのか、レート制限によるブロックなのか」を後で判別できるようにする
- 予備手段: 取得が不安定な場合は J-Quants API（JPX公式・無料プランあり）を検討
- **【追加】的中率評価バッチでも同じ`fetch_jp_stocks()`/`fetch_us_stocks()`を再利用し、
  評価対象銘柄のN営業日後の終値を取得する**

### 4-3. スクリーニング対象銘柄プール（動的スクリーニングの母集団）
市場全体（東証約4,000銘柄）を毎日スクリーニングするのはレート制限リスクが高いため、
以下の「業種別ウォッチリスト」を候補プールとして事前定義し、
その中から当日のトレンド判定結果に応じて動的に抽出する方式とする。

- 候補プールの例:
  - 半導体セクター: 日経半導体株指数の構成銘柄（東証上場、時価総額上位30銘柄、日本経済新聞社が
    毎年11月末に見直し）＋ SOX指数構成銘柄（米国、約30社）
  - 他業種を追加する場合は、TOPIX-17またはGICSセクターの代表銘柄リストを
    別途watchlist.csv等で管理する
- watchlist.csv（例）:
  ```
  ticker,sector,market
  6857.T,semiconductor,JP
  8035.T,semiconductor,JP
  NVDA,semiconductor,US
  AMD,semiconductor,US
  ```

## 5. ルール仕様（すべて閾値固定・ルールベース）

### 5-1. トレンド判定ルール（業種指数向け）
```python
def judge_trend(price_series, short=20, long=50):
    ma_short = price_series.rolling(short).mean().iloc[-1]
    ma_long  = price_series.rolling(long).mean().iloc[-1]
    momentum = price_series.iloc[-1] / price_series.iloc[-6] - 1  # 直近5営業日騰落率
    if ma_short > ma_long and momentum > 0.02:
        return "上昇トレンド"
    elif ma_short < ma_long and momentum < -0.02:
        return "下降トレンド"
    else:
        return "レンジ"
```
※ 閾値(0.02等)は仮の初期値。運用前に過去データでバックテストし、要調整。

### 5-2. RSI
- 期間14日、Wilderの平滑化法（`pandas_ta.rsi(close, length=14)`を使用）
- 70以上: 買われすぎ（分配段階の可能性） / 30以下: 売られすぎ

### 5-3. ワイコフ近似ルール（初期閾値、要バックテスト調整）
| 概念 | ルール化(例) |
|---|---|
| SC(セリングクライマックス) | 当日出来高が過去20日平均の2倍以上、値幅が過去20日平均の1.5倍以上、かつ陰線 |
| AR(オートマチックラリー) | SC後5営業日以内にSC安値から+5%以上の反発 |
| ST(セカンドテスト) | AR後、SC安値の±2%以内まで再下落後に反発 |
| Spring | 直近20日安値を1〜3%下回った後、3営業日以内にレンジ上限近くまで戻し、出来高は平均以下 |
| CHoCH | 直近スイング安値を切り上げる形成(Higher Low)の確認 |

### 5-4. 買い候補判定ロジック
`業種トレンド="上昇トレンド"` かつ `個別銘柄がSpring/CHoCHパターンに合致` かつ `RSI<70`
→ 買い候補アラート

### 5-5. 売り候補判定ロジック
`業種トレンド="下降トレンド"` または `保有銘柄が分配パターン(RSI≧70かつ出来高急増の陰線)に合致`
→ 売り候補アラート

## 6. アラート・通知送信仕様

### 6-1. リアルタイムアラート（従来仕様）
- 送信手段: **LINE Messaging API**（push message、自分専用のLINE公式アカウント経由）**＋メール（追加）**
- 必要情報: チャネルアクセストークン、送信先ユーザーID（LINE）、SMTP認証情報またはメール送信APIキー（メール）
  → すべてGitHub Secretsで管理し、リポジトリにハードコードしない
- 送信内容: 銘柄コード・銘柄名・判定理由（トレンド/RSI/ワイコフパターン）・判定日時
- LINEとメールは同一内容を並行送信する（どちらか一方が失敗しても他方は継続する独立実装とする）

### 6-2.【追加】メール送信の実装方針
- 実装手段（初期案・要確認）:
  - 個人利用の低頻度送信であるため、Python標準ライブラリの`smtplib`＋送信元Gmailアカウントの
    「アプリパスワード」を使う方式を第一候補とする（追加の有料サービス契約が不要なため）
  - 代替案として、SendGrid等のメール配信APIを使う方式も可能（Gmailの送信制限に抵触する
    ほど送信頻度が高い場合に検討）
  - 認証情報（送信元アドレス・アプリパスワード、または配信APIキー）はGitHub Secretsで管理
- 送信先: ユーザー本人のメールアドレス1件を`config.yaml`または環境変数で指定
- 件名の例: `【株式アラート】買い候補: 6857.T (2026-07-24)` /
  `【日次レポート】的中率サマリー (2026-07-24)`

## 7. 保有銘柄管理
- `holdings.csv` にてリポジトリ内で手動管理（銘柄コード、取得日、取得株数等）
- 売却が確定したら手動で該当行を削除する運用

## 8. エラーハンドリング要件
- データ取得失敗（空データ）は例外扱いにせず、ログに記録して処理を継続する
- LINE送信失敗時はリトライ1回、それでも失敗した場合はGitHub Actionsのログに残すのみとし、
  処理全体を止めない
- **【追加】メール送信失敗時も同様にリトライ1回のみとし、処理全体を止めない**
- **【追加】的中率評価バッチでの株価取得失敗（評価対象銘柄が上場廃止・データ欠損等）は、
  該当予測を「評価不能」としてスキップし、的中率の分母から除外する（無理に的中/不的中を
  判定しない）**

## 9.【追加】予測結果の的中率評価機能

### 9-1. 目的
日々発行される「買い候補」「売り候補」アラートについて、一定期間後の値動きを機械的に
判定し、ルールの有効性を継続的に可視化する。

### 9-2. 記録仕様
- アラート発生時に以下を`predictions.csv`（またはSQLite）へ1行追加する:
  ```
  prediction_id,date,ticker,signal_type,basis_price,eval_due_date,eval_price,result,status
  ```
  - `signal_type`: "buy" または "sell"
  - `basis_price`: アラート発生日の終値
  - `eval_due_date`: 判定対象日（`date` + N営業日後、Nは`config.yaml`で設定、初期値5営業日）
  - `eval_price`, `result`, `status`: 評価バッチ実行後に追記（`status`は"pending"→"evaluated"または"unavailable"）

### 9-3. 的中判定ルール（初期閾値、要バックテスト調整）
| signal_type | 的中条件（N営業日後の終値ベース） |
|---|---|
| buy | `eval_price >= basis_price × 1.02`（+2%以上） |
| sell | `eval_price <= basis_price × 0.98`（-2%以下） |

※ 閾値（N=5営業日、±2%）はいずれも仮の初期値であり、これが「正解」というものではありません。
運用しながらバックテストで妥当性を検証し、`config.yaml`で調整することを前提とします。

### 9-4. 的中率の集計・通知仕様
- 日次バッチで、`eval_due_date`が当日以前かつ`status="pending"`の予測を評価し、`status`を更新する
- 集計対象: 直近30件（または直近3ヶ月、`config.yaml`で選択可能）の評価済み予測
- 的中率 = 的中件数 ÷ 評価済み件数（"unavailable"は分母から除外）
- signal_type別（buy/sell）に的中率を分けて集計する
- 集計結果を日次メールで送信する（LINEへの送信は必須としない。文字数が多くなるためメールのみを既定とする）

### 9-5. 蓄積データの分析運用方針
- 本システムには、蓄積された`predictions.csv`をClaude(このチャット)が自動的に読み取り・
  分析する仕組みは含めない。的中率の集計・通知(9-4)まではシステムが自動で行うが、
  ルールの計算式そのものの改訂検討は自動化しない。
- 運用開始からある程度の期間(例:数ヶ月)が経過した段階で、ユーザーが手動で
  `predictions.csv`(または集計結果)をエクスポートし、Claudeのチャットにアップロードして
  分析を依頼する運用とする。
- Claude Codeはこの分析用の自動連携機能（例:GitHub ActionsからAPI経由でClaudeに
  自動送信する仕組み等）を、要求されない限り実装しない。

## 10. 重要な免責事項
- 本システムはルールベースの銘柄スクリーニング・通知・自己評価ツールであり、投資助言を行うものではない。
- ワイコフメソッドの近似ルール化は、あくまで実装上の一例であり、正解が存在するものではない。
  実運用前に必ず過去データでのバックテストを行うこと。
- 的中率評価の判定基準（N営業日、±2%等）も同様に仮の初期値であり、この数値自体の
  優劣を保証するものではない。
- 自動発注機能は含まない（発注はユーザー自身が証券会社アプリ等で手動実行する前提）。

# CLAUDE.md

このファイルはClaude Codeがこのリポジトリで作業する際に必ず読み込む規約ファイルです。
実装の詳細仕様は `要件定義書.md` を参照してください。このCLAUDE.mdはコーディング規約・
禁止事項・運用上の注意点を定義します。

## プロジェクト概要
日本株・米国株の業種特化型指数（SOX、日経半導体株指数等）のトレンドを毎日自動判定し、
ワイコフメソッドの近似ルールに合致する銘柄をLINEとメールにアラート送信するバッチシステム。
あわせて、過去のアラート（予測）が実際に的中したかどうかを日次で評価し、的中率をメール
サマリーとして送信する。GitHub Actionsのスケジュール実行で毎日自動稼働する。

## 絶対に守るべきルール

1. **AIの推論による判定を実装に含めない**
   すべての銘柄判定ロジック（トレンド判定、ワイコフ近似判定、RSI閾値判定、
   **的中/不的中の判定を含む**）は、`要件定義書.md` の5章・9章に記載された
   決定的な数値ルール（if-then/閾値比較）のみで実装する。
   LLM APIを呼び出して「この銘柄は買いか売りか」「この予測は的中したか」を
   AIに判断させるコードを書かない。

2. **秘密情報をコードにハードコードしない**
   LINE Messaging APIのチャネルアクセストークン、**メール送信用のSMTP認証情報
   （またはメール配信APIキー）**、データベンダーのAPIキー等は、
   すべて `os.environ` 経由で読み込み、GitHub Actionsの `secrets.*` から注入する。
   `.env` ファイルを使う場合は `.gitignore` に必ず追加する。

3. **自動発注コードを実装しない**
   本システムはアラート通知・的中率評価までが範囲。証券会社への自動発注（売買執行）
   ロジックは要求されない限り実装しない。

4. **yfinance利用時のレート制限対策を省略しない**
   個別銘柄データ取得は必ず20銘柄程度ずつのバッチ処理＋バッチ間1秒程度のスリープを入れる。
   取得結果が空の場合は例外を投げず、警告としてログに記録して処理を継続する。
   **的中率評価バッチ（`prediction_tracker.py`）が評価対象銘柄の株価を再取得する際も、
   同一のバッチ処理・スリープ規約を適用すること。**

5. **閾値はすべて設定ファイル（config）に外出しする**
   移動平均期間、モメンタム閾値、RSI閾値、ワイコフ近似ルールの数値基準、
   **的中率評価の判定期間（N営業日）や的中判定の閾値（±%）** などを
   コード内にマジックナンバーとして埋め込まず、`config.yaml` 等にまとめ、
   後から人間が調整しやすい構成にする。

6. **【追加】予測ログ(`predictions.csv`)は追記のみとし、過去の記録を上書き・削除しない**
   的中率の集計は「後から検証可能」であることが目的のため、評価済みレコードであっても
   `basis_price`や`date`等の元データを書き換えてはならない。評価結果(`eval_price`,
   `result`, `status`)のみを追記・更新可能とする。

7. **【追加】`predictions.csv`をClaudeへ自動送信・自動分析する仕組みを実装しない**
   ルールの計算式(閾値等)の改訂検討は、ユーザーが手動でデータをエクスポートし、
   別途Claudeのチャットに分析を依頼する運用とする(`要件定義書.md` 9-5参照)。
   GitHub ActionsからAPI経由でClaudeに自動送信する等の連携機能は、
   要求されない限り実装しない。

## リポジトリ構成（想定）
```
main.py                  … エントリーポイント、GitHub Actionsから呼ばれる
├─ data_fetcher.py        … 株価・出来高データ取得（yfinance、データベンダーAPI呼び出し）
├─ indicators.py          … RSI・移動平均等の計算（pandas-ta利用）
├─ trend_rules.py         … 業種指数のトレンド判定ルール
├─ wyckoff_rules.py       … ワイコフ近似ルール（SC/AR/ST/Spring/CHoCH判定）
├─ screener.py            … トレンド判定とワイコフ近似ルールを組み合わせた銘柄スクリーニング
├─ notifier.py            … LINE Messaging APIへの通知送信
├─ email_notifier.py      … 【追加】メール送信（アラート通知＋日次的中率サマリー）
├─ prediction_tracker.py  … 【追加】予測の記録・的中判定・的中率集計
├─ config.yaml            … 閾値・スケジュール・的中率評価パラメータ等の設定値
├─ watchlist.csv          … スクリーニング対象の候補銘柄プール（業種別）
├─ holdings.csv           … 保有銘柄リスト（手動管理）
├─ predictions.csv        … 【追加】予測ログ（追記専用、的中率評価の元データ）
└─ .github/workflows/
    ├─ daily_screening.yml … スクリーニング・アラート送信（8:00実行）
    └─ daily_evaluation.yml … 【追加】的中率評価・サマリーメール送信（8:10実行）
```

## コーディングスタイル
- Python 3.11以上を想定
- 型ヒントを付ける（`def judge_trend(price_series: pd.Series) -> str:` のように）
- 外部API呼び出し箇所は必ず `try/except` で囲み、失敗時のログ出力を行う
- テスト用に、実際のAPI呼び出しをモック化した単体テストを `tests/` に配置することを推奨
- **`prediction_tracker.py`の的中判定関数は、`trend_rules.py`等と同様に純粋関数として実装し、
  単体テストで境界値（ちょうど+2%等）の挙動を検証できるようにする**

## 実装時に確認すべき既知の制約
- LINE Notifyは2025年3月31日に終了済み。**LINE Messaging APIのみを使用する**こと
  （古いチュートリアル記事にLINE Notifyのサンプルコードが多数残っているため、
  実装時にAIが古い情報を参照して誤ってLINE Notify用のコードを生成しないよう注意する）。
- SBI証券は個人向けの汎用APIを提供していない。データ取得にSBI証券APIを使うコードを
  実装しようとしない。
- yfinanceは非公式ライブラリのため、Yahoo Finance側の仕様変更で突然動作しなくなる
  可能性がある。バージョンを `requirements.txt` に固定して明記する。
- **【追加】メール送信をGmailのSMTP（アプリパスワード方式）で実装する場合、通常のログイン
  パスワードでは認証できない点に注意する（2段階認証を有効化したうえでの専用アプリパスワード
  が必要）。この点はGoogle側の仕様であり、本プロジェクト固有の設定ではないため、実装前に
  ユーザー自身のGoogleアカウント設定画面で最新の手順を確認すること。**
- **【追加】`predictions.csv`のレコード数は運用が長期化すると増大するため、読み込み時に
  全件をメモリに載せる設計で問題ないか、件数が数万行を超えた場合はSQLite等への移行を
  検討する旨をコメントで残しておく（初期実装ではCSVで問題ない）。**

## 作業の進め方
1. `要件定義書.md` を読み込み、全体設計を把握する（**特に9章の的中率評価仕様を確認**）
2. `config.yaml` と `watchlist.csv` の雛形を作成する（**的中率評価用のN営業日・閾値パラメータを含める**）
3. `data_fetcher.py` → `indicators.py` → `trend_rules.py` → `wyckoff_rules.py` →
   `screener.py` → `notifier.py` → `email_notifier.py` → `prediction_tracker.py` の順に
   実装し、各モジュール単体で動作確認する
4. `main.py`（スクリーニング＋アラート）と、**別エントリーポイントとしての評価バッチ**
   （的中率評価＋サマリーメール送信）を結合し、GitHub Actionsのワークフローファイルを
   2本（`daily_screening.yml`、`daily_evaluation.yml`）作成する
5. 実装が要件定義書の閾値・ルールと一致しているか、実装後に差分を確認する
6. **【追加】`predictions.csv`にテスト用のダミーレコードを数件投入し、`prediction_tracker.py`が
   正しく的中/不的中/評価不能を判定できるかを確認する**

# 個別株スクリーニング&アラートシステム 要件定義書

## 0. 前提・重要な訂正
- 当初検討していた「LINE Notify」は2025年3月31日にサービス終了済み。
  本システムでは後継の **LINE Messaging API** を使用する。
- 個人が自分専用に通知を受け取る用途のため、LINE公式アカウントを自分で作成し、
  自分自身（または非公開のLINEグループ）にpushメッセージを送る構成とする。
  Messaging APIは月200通まで無料（個人利用の通知回数なら十分収まる想定）。

## 1. 目的
海外業種特化型指数（例: SOX等）と国内業種特化型指数（例: 日経半導体株指数）の
トレンドを毎日自動判定し、ワイコフメソッドの近似ルールに合致する銘柄を
「買い候補」「売り候補」としてLINEにアラート送信する。
判定はすべてルールベース（閾値による決定的ロジック）で行い、
AIの都度推論には依存しない。

## 2. システム全体構成
```
[①データ取得層] → [②ルール判定エンジン層] → [③スケジューラ(GitHub Actions)] → [④アラート層]
```
- 実行環境: GitHub Actions（スケジュール実行 = cron構文）
- 言語: Python 3.x
- 主要ライブラリ: yfinance, pandas, pandas-ta, requests

## 3. スケジュール仕様
| 時刻(JST) | 処理内容 |
|---|---|
| 5:30頃 | 米国業種特化型指数のトレンド判定（前日の米国市場クローズ後データを使用） |
| 8:00 | ①の結果を使い、上昇トレンド業種内の候補銘柄をワイコフ近似ルールでスクリーニング→買い候補アラート |
| 8:00 | 下降トレンド業種＋保有銘柄リストをワイコフ近似ルールで判定→売り候補アラート |

- GitHub Actionsの `schedule: cron:` はUTC基準のため、JST変換時に夏時間・冬時間の概念がない点に注意
  （日本はDSTがないため一定のUTCオフセット+9時間で固定でよい。米国側の夏時間切替は
  「米国市場クローズ後のデータを使う」設計のため、こちらで時刻補正する必要はない）。

## 4. データ取得仕様

### 4-1. 米国業種特化型指数（トレンド判定用）
- 取得元: 海外データベンダー（既定：使用するベンダーのAPIキーは環境変数/GitHub Secretsで管理）
- 対象指数（例）: SOX, XLK, XLV, XLF, XLE, XLU等（別紙「業種特化型指数一覧」参照）

### 4-2. 個別銘柄データ（日本株・米国株、RSI・ワイコフ判定用）
- 取得元: **yfinance**（無料・APIキー不要、非公式ラッパーである点に留意）
- 日本株ティッカー: 証券コード+".T"（例: "6857.T"）
- 米国株ティッカー: そのまま（例: "NVDA"）
- 取得は20銘柄ずつバッチ処理し、バッチ間に1秒程度のインターバルを入れる
- 取得結果が空(`df.empty`)の場合はエラーではなく警告としてログに記録し、
  「データ欠損なのか、レート制限によるブロックなのか」を後で判別できるようにする
- 予備手段: 取得が不安定な場合は J-Quants API（JPX公式・無料プランあり）を検討

### 4-3. スクリーニング対象銘柄プール（動的スクリーニングの母集団）
市場全体（東証約4,000銘柄）を毎日スクリーニングするのはレート制限リスクが高いため、
以下の「業種別ウォッチリスト」を候補プールとして事前定義し、
その中から当日のトレンド判定結果に応じて動的に抽出する方式とする。

- 候補プールの例:
  - 半導体セクター: 日経半導体株指数の構成銘柄（東証上場、時価総額上位30銘柄、日本経済新聞社が
    毎年11月末に見直し）＋ SOX指数構成銘柄（米国、約30社）
  - 他業種を追加する場合は、TOPIX-17またはGICSセクターの代表銘柄リストを
    別途watchlist.csv等で管理する
- watchlist.csv（例）:
  ```
  ticker,sector,market
  6857.T,semiconductor,JP
  8035.T,semiconductor,JP
  NVDA,semiconductor,US
  AMD,semiconductor,US
  ```

## 5. ルール仕様（すべて閾値固定・ルールベース）

### 5-1. トレンド判定ルール（業種指数向け）
```python
def judge_trend(price_series, short=20, long=50):
    ma_short = price_series.rolling(short).mean().iloc[-1]
    ma_long  = price_series.rolling(long).mean().iloc[-1]
    momentum = price_series.iloc[-1] / price_series.iloc[-6] - 1  # 直近5営業日騰落率
    if ma_short > ma_long and momentum > 0.02:
        return "上昇トレンド"
    elif ma_short < ma_long and momentum < -0.02:
        return "下降トレンド"
    else:
        return "レンジ"
```
※ 閾値(0.02等)は仮の初期値。運用前に過去データでバックテストし、要調整。

### 5-2. RSI
- 期間14日、Wilderの平滑化法（`pandas_ta.rsi(close, length=14)`を使用）
- 70以上: 買われすぎ（分配段階の可能性） / 30以下: 売られすぎ

### 5-3. ワイコフ近似ルール（初期閾値、要バックテスト調整）
| 概念 | ルール化(例) |
|---|---|
| SC(セリングクライマックス) | 当日出来高が過去20日平均の2倍以上、値幅が過去20日平均の1.5倍以上、かつ陰線 |
| AR(オートマチックラリー) | SC後5営業日以内にSC安値から+5%以上の反発 |
| ST(セカンドテスト) | AR後、SC安値の±2%以内まで再下落後に反発 |
| Spring | 直近20日安値を1〜3%下回った後、3営業日以内にレンジ上限近くまで戻し、出来高は平均以下 |
| CHoCH | 直近スイング安値を切り上げる形成(Higher Low)の確認 |

### 5-4. 買い候補判定ロジック
`業種トレンド="上昇トレンド"` かつ `個別銘柄がSpring/CHoCHパターンに合致` かつ `RSI<70`
→ 買い候補アラート

### 5-5. 売り候補判定ロジック
`業種トレンド="下降トレンド"` または `保有銘柄が分配パターン(RSI≧70かつ出来高急増の陰線)に合致`
→ 売り候補アラート

## 6. アラート送信仕様
- 送信手段: **LINE Messaging API**（push message、自分専用のLINE公式アカウント経由）
- 必要情報: チャネルアクセストークン、送信先ユーザーID（すべてGitHub Secretsで管理し、
  リポジトリにハードコードしない）
- 送信内容: 銘柄コード・銘柄名・判定理由（トレンド/RSI/ワイコフパターン）・判定日時

## 7. 保有銘柄管理
- `holdings.csv` にてリポジトリ内で手動管理（銘柄コード、取得日、取得株数等）
- 売却が確定したら手動で該当行を削除する運用

## 8. エラーハンドリング要件
- データ取得失敗（空データ）は例外扱いにせず、ログに記録して処理を継続する
- LINE送信失敗時はリトライ1回、それでも失敗した場合はGitHub Actionsのログに残すのみとし、
  処理全体を止めない

## 9. 重要な免責事項
- 本システムはルールベースの銘柄スクリーニング・通知ツールであり、投資助言を行うものではない。
- ワイコフメソッドの近似ルール化は、あくまで実装上の一例であり、正解が存在するものではない。
  実運用前に必ず過去データでのバックテストを行うこと。
- 自動発注機能は含まない（発注はユーザー自身が証券会社アプリ等で手動実行する前提）。
